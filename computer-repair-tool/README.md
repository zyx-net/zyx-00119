## 快速验证：报价改价 Bug 修复

> **Bug 描述**：同一工单出过一次报价后，修改配件/工时价格再生成新报价，金额仍沿用旧价。
>
> **根因**：`showQuoteModal` 用旧报价的历史价格填充输入框，设置 `select.value` 不触发 `onchange`，导致价格停留在旧值。
>
> **修复（2处，双重保险）**：
> 1. **UI层**：预填充后手动调用 `quotePartChange`/`quoteLaborChange`，从 select 的 `data-price`（最新配置）重算
> 2. **数据层**：`doQuote` 保存时根据 `partId` 从 `Store.getParts()` 实时读最新价，不依赖 DOM

### 一键验证（推荐，30秒出结果）
1. 启动本地服务器（见上方）
2. 浏览器打开 `http://localhost:8080/test-quote-price.html`
3. 点击 **「▶ 运行全部测试」**
4. 看到 **「🎉 全部测试通过（8/8）」** 即表示修复正确

测试覆盖：改价后二次报价、旧版本不串改、刷新一致性、多版本独立、详情页渲染正确等8个用例。

**核心原则：一套用例定义，三处入口执行，结果格式完全统一，跨重启可恢复，失败可追溯到步骤。**

三端（浏览器/Node.js/Python）使用完全一致的：

- **必填字段**：`runId`/`suiteVersion`/`startedAt`/`total`/`passed`/`failed`/`results`

### 完整链路说明

**执行 → 持久化 → 恢复 → 对比 → 导出 → 导入** 全链路：

1. **执行**：运行测试，逐条记录断言步骤，失败时捕获完整执行轨迹
2. **持久化**：结果自动保存到 `crp_test_results`，页面状态保存到 `crp_test_state`
3. **恢复**：刷新/重启/重新打开页面后，自动检测并提示恢复上次结果
4. **对比**：历史记录中任意两次运行可对比，识别回归和修复
5. **导出**：选择导出全部或单次运行，包含完整日志和步骤信息
6. **导入**：导入时自动去重、校验字段、检测冲突，**冲突时不覆盖旧记录**，保留完整历史

### 验收标准（三端一致）

运行以下三段流程，确认展示、历史和提示完全一致：

1. **执行测试**：12 个用例全部通过，结果保存成功

3. **导出再导入**：导出结果 → 清空历史 → 导入文件 → 确认数据完整恢复，无重复、无覆盖、无丢失

| **待审区** | 上传文件→预检→确认写入的完整待处理流程 |
| **权限配置** | 角色切换、权限矩阵管理、处理人增删改 |

#### 2. 详情页五种子 Tab
每个批次详情页包含 5 个信息视图：
- **摘要信息**：成功摘要卡片 + 预检5类问题统计
- **冲突项**：按严重度排序展示5类冲突（权限拦截 > 缺字段 > 关联缺失 > 版本不匹配 > 重复）
- **导入明细**：每条导入记录的类型、ID、状态、原因、时间戳
- **时间线**：创建→执行→完成→回滚等全流程节点
- **失败日志**：失败项列表、失败统计、跳过项明细

### 5 类预检问题分类

| # | 冲突类型 | 严重度 | 说明 |
|---|---------|--------|------|
| 1 | **无权限** | `blocked` 🔴 | 当前角色对该数据类型无导入权限，**直接拦下** |
| 2 | **缺少字段** | `error` 🔴 | 必填字段缺失或为空，该记录无法导入 |
| 3 | **关联缺失** | `error` 🔴 | 报价/历史关联的工单不存在 |
| 4 | **版本不匹配** | `warning` 🟡 | 同一工单相同版本报价已存在 |
| 5 | **重复数据** | `warning` 🟡 | ID 或工单号已存在 |

**重要保证**：
- 出现 `blocked`（无权限）时，**整个批次无法确认导入**
- 其余冲突项按「不覆盖」原则处理，自动跳过并记录
- 所有问题在预检结果中**分类列出**，不混在一起

### 权限规则系统

#### 三种角色
| 角色 | 说明 |
|------|------|
| `admin` | 超级管理员，全部权限 |
| `operator` | 操作员，可导入、可回滚、可导出 |
| `viewer` | 仅查看，任何写入都被拦截 |

#### 权限矩阵
| 权限项 | admin | operator | viewer |
|--------|:-----:|:--------:|:------:|
| 导入工单 | ✅ | ✅ | ❌ |
| 导入报价 | ✅ | ✅ | ❌ |
| 导入历史 | ✅ | ✅ | ❌ |
| 按批次撤销 | ✅ | ✅ | ❌ |
| 冲突回退 | ✅ | ✅ | ❌ |
| 导出结果 | ✅ | ✅ | ✅ |
| 修改权限配置 | ✅ | ❌ | ❌ |

- 权限配置保存在 `localStorage` 的 `crp_permission_config`

- 浏览器重启后配置**完全生效**，无需重新设置

#### 处理人管理
- admin 可增删处理人（姓名唯一）
- 导入、回滚、撤销操作必须选择**有效处理人**
- 非法处理人会被拦截并给出明确提示

### 批次处理系统

#### 批次号规则
```
格式：BATCH-YYYYMMDD-NNNN
示例：BATCH-20260618-0001
```

- 批次号全局唯一

- ✅ **失败日志**：失败项 ID、类型、原因、时间

- ✅ **时间线**：全流程操作节点
- ✅ **异常提示文案**：可读性友好的错误说明

### 撤销与回放功能

#### 1. 按批次撤销
- 一键恢复到导入前的完整状态
- 回滚需校验权限（viewer 禁止）
- 已撤销的批次不可重复撤销
- 撤销记录写入时间线，包含处理人和原因

#### 2. 冲突回退
- 仅回退指定类型的冲突项（如仅回退重复数据）
- 其余正常导入数据保持不变
- 适合部分数据需要保留的场景

#### 3. 导出后再导入校验
- 导出的批次结果可重新导入系统进行校验
- 校验内容包括：
  - 格式版本兼容性
  - 数据完整性
  - 批次号唯一性
  - 与现有数据的一致性
- 校验结果给出详细报告

### 完整操作流程

#### 标准导入流程（待审区）
1. 点击左侧导航「🛡️ 导入准入与回放台」
2. 切换到「待审区」Tab
3. 点击上传区域选择 JSON 文件
4. **自动预检**：查看 5 类问题分类结果
5. 🔴 如存在「无权限」拦截 → 无法继续，换角色或联系管理员
6. 选择**处理人**（必填），填写操作备注（可选）
7. 点击「✅ 确认写入」
8. 等待执行完成，自动跳转到批次详情
9. 查看成功摘要、冲突明细、失败日志、时间线

#### 按批次撤销流程
1. 在批次列表找到目标批次（状态：completed）
2. 点击「↩ 按批次撤销」
3. 填写处理人和撤销原因
4. 确认后数据恢复到导入前，状态变为「已撤销」

#### 冲突回退流程
1. 打开批次详情 → 冲突项 Tab
2. 点击「⚙️ 冲突回退」
3. 选择需要回退的冲突类型
4. 填写处理人和原因
5. 执行后仅回退指定类型冲突项

#### 导出后再导入校验
1. 批次详情 → 点击「📤 导出结果」下载 JSON
2. 点击「🔍 导出再导入校验」
3. 选择刚下载的 JSON 文件
4. 查看校验报告：格式/版本/数据/一致性检查结果

#### 权限配置流程
1. 切换到「权限配置」Tab
2. **切换当前角色**：点击角色按钮（admin/operator/viewer）
3. **修改权限矩阵**：勾选/取消勾选各权限项（仅 admin 可改）
4. **管理处理人**：输入姓名添加，点击删除移除
5. 所有修改自动保存，重启后生效

### 常见问题排查

#### Q1：上传文件后提示「无权限导入工单」
- **原因**：当前角色（如 viewer）无对应数据类型导入权限
- **解决**：切换到 admin 或 operator 角色，或在权限配置中开通权限

#### Q2：处理人下拉为空
- **原因**：处理人列表为空
- **解决**：admin 角色进入「权限配置」Tab → 添加处理人

#### Q3：批次执行后数据数量偏少
- **查看**：批次详情 → 导入明细 / 失败日志
- **常见原因**：重复数据、版本冲突被自动跳过；缺字段、关联缺失被标记错误

#### Q4：回滚按钮灰色不可点
- **原因**：批次已撤销 / 权限不足 / 批次未完成
- **检查**：批次状态是否为 completed，当前角色是否允许回滚

#### Q5：重启后筛选条件没恢复
- **检查**：浏览器是否禁用 localStorage
- **检查**：是否使用无痕/隐私模式
- **检查**：是否清理过浏览器数据

### 数据安全保证
1. **本地存储**：所有数据仅保存在浏览器 localStorage
2. **快照机制**：导入前保存数据快照，确保可完整回滚
3. **不覆盖原则**：冲突数据坚决不覆盖，保护现有数据
4. **权限拦截**：无权限操作在预检阶段直接拦下
5. **建议备份**：大批量导入前先导出原有数据备份
